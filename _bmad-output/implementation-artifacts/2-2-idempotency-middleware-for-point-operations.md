# Story 2.2: Idempotency Middleware for Point Operations

Status: done

## Story Narrative

As a Developer,
I want to implement idempotency middleware for mutating API endpoints,
So that duplicate requests from network retries do not cause double-spending.

## Acceptance Criteria

- **Given** a request to `/api/transactions/*` or `/api/withdrawals/*`
- **When** an `Idempotency-Key` header is provided
- **Then** the `idempotency.ts` middleware verifies the key in MongoDB (AD4)
- **And** it rejects duplicates within a 24-hour window by returning the cached original response (NFR-SEC-2).

## Developer Context

### Technical Requirements

- **Idempotency Key Model:** `packages/db/src/models/idempotency.model.ts`
  - `_id`: String (auto)
  - `key`: String, unique, required
  - `response`: Mixed (cached response body)
  - `statusCode`: Number
  - `createdAt`: Date (TTL index: expires after 86400 seconds / 24 hours)
- **Express Middleware:** `apps/server/src/middleware/idempotency.ts`
  - Must intercept requests, check the `Idempotency-Key` header.
  - If no header, reject with 400 Bad Request.
  - If key exists in DB, immediately return the cached `statusCode` and `response` and halt further processing.
  - If key is new, capture the response using techniques like overriding `res.json` and `res.send`. Once the response is generated by the downstream handlers, save the key, `statusCode`, and `response` to MongoDB.
- **Error Handling:** Avoid caching 5xx errors if possible, as a retry might succeed if the server recovers, but MUST cache 4xx errors if they are deterministic business logic failures (e.g., insufficient balance).

### Architecture Compliance

- **AD4 (Idempotency Keys):** All mutating endpoints (transfer, withdraw, distribute) require an `Idempotency-Key` header (client-generated UUID). The server stores used keys in a MongoDB collection with a 24-hour TTL index.
- **NFR-SEC-2:** The system must reject duplicate transaction requests containing the same idempotency key within a 24-hour window.

### File Structure Requirements

- **UPDATE** `packages/db/src/index.ts` to export the new model.
- **CREATE** `packages/db/src/models/idempotency.model.ts`
- **CREATE** `apps/server/src/middleware/idempotency.ts`

### Testing Requirements

- Unit tests for the middleware behavior (e.g., matching cached responses, handling missing headers).
- Validation that the TTL index is correctly defined on the Mongoose schema.

## Previous Story Intelligence (from Story 2.1)

- The project relies on Mongoose `session.withTransaction()` for atomic operations.
- The `IdempotencyKey` check happens *before* the transaction starts in the route handler, preventing unnecessary database load and potential deadlock issues during retries.
- The `transaction.model.ts` also has a unique, sparse `idempotencyKey` index, adding a second layer of defense, but the middleware is the primary mechanism for returning the *cached response*.
